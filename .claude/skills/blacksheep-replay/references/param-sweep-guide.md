# パラメータスイープ設計ガイド

## スイープの目的

スイープは「どのパラメータがどの方向に影響するか」を把握するためのもの。
全組み合わせを網羅するのではなく、**感度の高い軸を特定→絞り込み**という 2 段階が効率的。

---

## 段階 1: 粗いグリッド（5〜9 セル）

まず主要パラメータを 3 値ずつ試す。

```
mean_reversion_01 例:
  window: [5, 10, 20]   ← 3 値
  k:      [1.0, 2.0]    ← 2 値（計 6 セル）
  holding_minutes: [30] ← 固定
```

判断:
- PnL が `window` に大きく依存 → Stage 2 で `window` を細かく刻む
- PnL が `k` に大きく依存 → Stage 2 で `k` を細かく刻む
- 全セルが似たような結果 → パラメータより期間/銘柄の問題かもしれない

---

## 段階 2: 精密スイープ（感度が高い軸のみ）

段階 1 でベストが `window=10, k=1.5` 付近なら:

```
  window: [8, 10, 12, 15]
  k:      [1.2, 1.5, 1.8]
  holding_minutes: [20, 30]
```

これで 4×3×2 = 24 セル程度。

---

## 採用閾値の決め方

```
baseline_best_pnl = 既存 Silver の最高 total_pnl
採用ライン = baseline_best_pnl × 1.0  # 同等以上
stretch 目標 = baseline_best_pnl × 1.5  # 1.5 倍
```

drawdown 上限は「許容できる最大損失 / initial_cash」で決める。
デフォルト: 初期キャッシュの 20%（1M JPY の場合: -200,000 JPY）。

---

## 注意事項

- **過学習に注意**: 特定期間で最適化したパラメータは他の期間で逆効果になりやすい。
  スイープ後に「Out-of-Sample 期間」で検証することを推奨。
- **セル数上限**: 25 セルを超えると replay 時間が 30 分以上かかる。
  並列実行は e-station が in-process モードで 1 プロセスしか走らせられないため非推奨。
- **最低 trade_count**: 統計的意味のある評価には trade_count ≥ 5 が必要。
  エントリ条件が厳しすぎる場合は `k` を緩める。
- **日付範囲の検証**: スイープと同じ期間でパラメータ最適化したものを「最適」と言うのは
  in-sample bias。可能であれば別期間で検証する。
