## 0. これは何をするものか / なぜデモでやるのか

`note「寄付き板寄せ同業ペア乖離3分回帰」` の凍結15ペアを、**前場の寄付き板寄せで両脚建て → 各脚を実約定から180秒保有 → 成行返済**する。デモ口座（検証環境）に実発注してフォワードテストする。

**重要な背景**: 2日プローブの `+12.54bps` は、出口を「固定 09:00:00」で測ったことによる計測アーティファクトだった。遅延寄り銘柄では固定アンカーの実質保有が0秒になり損失が消えていた。**実約定タイミングで測り直すと同じ2日は −27〜−38bps の負け**（07-13 は寄り付きに80秒の録画ギャップもあり評価不能）。

実発注は timing を強制的に正直にする:

- entry = **各脚が実際に寄った瞬間の約定値**（day_open）
- exit  = **その脚の実約定時刻 + 180秒**（固定09:00ではない）

したがってこのハーネスの記録（`logs/*.jsonl` の `SUMMARY.mean_net_bps`）は、**アーティファクト抜きの本当の期待値**になる。これがデモでフォワードテストする最大の意義。

> ⚠️ 期待値: 過去2日の正直な計測では負け。デモでも負ける可能性が高い。**これは「戦略を採用するため」ではなく「計測を正す・仮説を殺すか生かすか決める」ためのテスト**。損益が出たら勝ちではなく、`kill 基準`（§7）に照らして判定する。

## 7. 結果の読み方と撤退条件

ログは `live/logs/pair_open_fade_YYYY-MM-DD.jsonl`。各行が1イベント（`signals`, `entry_filled`, `exit_sent`, `leg_result`, `SUMMARY`）。

**最重要は最後の `SUMMARY.mean_net_bps`**（全脚の net bps 平均）と、各 `leg_result`。

判定は note の凍結ルールに従う。**10営業日を仕様固定で溜めてから**、次のどれかで棄却:

1. 180秒の出口摩擦控除後の日次平均 net が 0 以下
2. 正の日が 10日中 7日 未満
3. 各脚 +2bps ストレスで net が 0 以下
4. 最大1ペア / 1日 / 1業種 を除くと符号反転
5. **寄付きの非約定・部分約定を含む実現可能性が、想定建玉の80%未満**
   → デモでは `entry_filled` が出た脚数 ÷ 発注脚数で直接測れる。これが本テストの主目的の一つ。

**このフォワードテストでしか分からないこと**（バックテストで捏造された点）:

- 遅延寄り（特別気配で寄らない）銘柄が実際どれだけ出るか、寄成が本当に約定するか（kill #5）
- 各脚を実約定から180秒持ったときの本当の net（固定アンカーではない）
- ペアの両脚が「同時」に建つのか、数分ずれるのか（電線 5801/5802/5803 は過去2日ともずれた）

## 8. 安全上の注意 / 既知の落とし穴

- **必ず 18081（検証）とドライランから**。`PORT=18080`（本番）に変えるのは、デモで一連が正しく通ることを確認した後だけ。
- **アプリのログイン維持**: kabu ステーションが落ちると API も落ちる。寄り前に起動確認。
- **レート制限**: 発注は 0.3秒間隔、板は 2秒間隔に絞ってある。エラーが出たら間隔を広げる。
- **重複銘柄**: 同じ銘柄が複数ペアに出る（例 8306 は4ペア、8035 は4ペア）。本ハーネスは
  **重複銘柄を建て増ししない**（最初に決まった方向で1建玉）。note のペア独立損益とは厳密には
  一致しないが、デモの単純化として許容。厳密なペア独立が要るなら建玉を分けて `HoldID` で
  管理するよう拡張する。
- **返済の HoldID**: `send_close` は `positions` から取った `HoldID` で建玉を指定して返済する。
  デモで `positions` のフィールド名（`HoldID` / `ExecutionID`）が違う場合はコードの該当箇所を直す。
- **約定時刻の近似**: 建玉検知は `positions` のポーリング（1秒間隔）なので、実約定から検知まで
  最大1秒の遅れがある。180秒ホライズンに対しては無視できる誤差。
- **DRY_RUN の SUMMARY**: ドライランでは実約定値が無いため bps は出ない（配管確認専用）。
