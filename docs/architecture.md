# Architecture

このプロジェクトは `backcast` を外部インフラとして参照し、日中の板情報から自動売買戦略を作るためのワークスペースです。

## Constraints

- `C:\Users\sasai\Documents\backcast` は変更しません。
- 戦略はデイトレードに限定します。翌営業日へポジションを持ち越す前提の処理は gold layer 以降に入れません。
- 入力元は `S:\jp\stocks_board_kabu_push` の板情報です。

## Layers

### Bronze

入力ファイルを原本に近い形で保存します。監査性を優先し、この層ではできるだけ解釈を加えません。

### Silver

板イベントを正規化します。主な想定カラムは `ts_event`, `symbol`, `bid_price`, `bid_size`, `ask_price`, `ask_size`, `trade_price`, `trade_size` です。

### Gold

戦略が読む特徴量を保存します。例はスプレッド、板厚、短期インバランス、約定方向、セッション内累積出来高、シグナル、バックテスト結果です。

gold の `target` は `signals.fold_states` を Python で畳み込んで書き出します（SQL で書き直しません）。
バックテストと gold が同じ関数を通るため、両者のルールが食い違うことは構造的に起こりません。

## Backtest: セッションは互いに独立

`run_backtest` は tick を**日単位のセッション**に分割し（`prepare`）、セッションごとに
独立して回します。各セッションの最終 tick でポジションを強制フラット（`min_hold_secs` より
優先）、シグナルの状態・約定ディレイの窓・秒時計もすべて翌朝リセットされます。
夜をまたいで持ち越されるものは何もありません。

これは `--date` を省いて複数日をまとめて回すとき（`date=*` glob）に効きます。日次の
強制フラットが無いと、最終 tick まで持ち越した「翌日跨ぎポジション」が損益に混入します。

## Signal: churn 抑制と、その限界

`signals.py` は「板 imbalance → 目標ポジション」を決める**純粋な状態機械**です。I/O を持たず、
backcast の marimo cell（live）と同じ関数を共有できます。状態（平滑化平均・現在の目標・
最終変更時刻）は呼び出し側が持ち、`update(state, ts, imbalance, params) -> SignalState` は
純粋関数のままです。

3 つの churn 抑制機構を持ちます。

1. **ヒステリシス** — `enter_threshold` で建て、`exit_threshold`（≤ enter）を割るまで持つ。
   ドテンは「新規建て」と同じ扱いで、反対側の `enter_threshold` が必要です。
2. **平滑化** — 各観測が単位質量を持つ時間減衰平均。等間隔なら教科書どおりの EMA
   (`d*mean + (1-d)*x`) に一致しつつ、この板データ特有の 2 つの病理も扱えます:
   同一タイムスタンプ（全体の 2.8%）でも新しい観測が正しい重みで入り、
   昼休みのような長い空白では古い質量が減衰して平均が作り直されます。
3. **最小保有 / クールダウン** — 建てたら `min_hold_secs`、閉じたら `cooldown_secs` は
   目標を変えない。時計は**約定時刻ではなく判断時刻**から回します（live のシグナルは
   自分の約定を同期的に知れないため、バックテストもそれを覗いてはいけない）。

### 2026-07-09（9984 / 285A / 5803）の検証結果

churn 抑制そのものは効きます。`CHURN_CONTROLLED` プリセット
（enter=0.80, exit=0.00, halflife=1.0s）は約定を **278 分の 1** に減らし、
baseline の損失 -64.96M JPY のうち **99.93%** を消しました（-43,032 JPY）。

**しかしこの戦略は成行では原理的に勝てません。**

- imbalance は確かに予測力を持ちます（1〜20 tick 先の mid を単調に予測）。
  しかしピークのドリフトは 9984 で +0.195 円、確信度を上げて `imbalance > 0.9` にしても +0.308 円。
- 一方、往復コストは **スプレッド 1.20 円 + 手数料 1.74 円 = 2.95 円/株**（9984, 1.5bps）。
- つまり **edge / cost ≈ 0.07 倍**。最も条件の良い `imbalance > 0.9` でも
  ドリフトはスプレッドの **0.26 倍**にしかなりません。
- 手数料を **0 にしても勝てません**: 往復 100 回以上を取る 193 通りの設定のうち、
  gross（スプレッド支払い後・手数料無し）がプラスになったものは **0 件**でした。
- グリッドを net で並べると最適解は「ほとんど取引しない」隅に落ちます（往復 1〜5 回で
  net > 0 に見えるのは単なるノイズ）。銘柄をまたぐ OOS でも転移しません
  （285A で fit した設定は 9984 で -17,736 JPY）。

したがって churn 抑制は net を **0 に近づける**だけで、edge を作り出しません。
`sweep` が `net/trip` 列を出すのはこのためです。**往復あたりの損益が正でない限り、
どれだけ約定を減らしても勝ちにはなりません。**

次に効くのは指値（passive）約定でスプレッドを**払う側から受け取る側**に回ることですが、
backcast の live 発注は現状 `submit_market` の成行のみです。ここが本当のボトルネックです。

## Maker: 保守的 passive-fill シミュレータと、その検証結果（2026-07-10）

taker の結論を受けて、指値でスプレッドを受け取る側に回る **maker シミュレータ**
（`maker.py` / `maker_strategies.py`）を作り、fill-adjusted expected value を直接測った。

### 約定モデル（保守的に倒してある）

板 push データには歩み値が無い。約定は連続するスナップショット間の累積 `volume` の差分と
`last_px` から推定する。ルールは事前登録してから実装した:

1. **Queue**: 自分の指値の前には「到着時点で同値に表示されている全数量」が並ぶ。
   前が減るのは**推定約定分だけ**。表示数量の減少（＝前方のキャンセル）では前に進まない。
2. **Through-fill**: 自分の価格より不利な側で約定が印字されたら、価格優先の原理により全量約定。
3. **Crossing**: 反対側の気配が自分の価格を跨いで表示されたら約定（自分の指値が生きたまま
   板がクロスすることはあり得ない）。
4. **Latency**（既定 0.5s）: 発注・キャンセル・**約定通知**のすべてに片道遅延。キャンセルは
   約定に負ける。板が遅延中に動いて成行同然で到着した指値は「自分の指値価格で」taker 約定
   （現実には touch で約定するので悲観方向）。
5. **Auction 除外**: 特別気配（sign 0107）と [11:29–12:31]・[15:24–] の窓では約定・queue 前進
   ともに発生しない（11:29:59 / 12:29:59 / 15:29:59 の巨大 itayose 印字が queue を進めない）。
   窓に入ると指値は自動キャンセル。
6. **日次セッション独立**: taker backtest と同じ不変条件。15:24 直前の最終连続気配 snap で
   強制成行フラット。
7. **PnL 恒等分解**: net = spread_capture + taker_edge + adverse_selection(≤10s)
   + inventory_drift − commission が**厳密に**成り立つ（テストで検証）。

fill モデルの健全性ゲート（無条件 join-the-bid ベンチマーク、手数料 0）:
9984 で **-114 円/往復**、6834 で **-4,564 円/往復**。adverse selection が支配的で、
「タダで儲かる fill モデル」ではないことを確認してから戦略を評価した。

実装後の code review で結果を歪める実バグを 2 件検出・修正した（TakerExit の意図が
非取引窓の間に積み上がり再開時に全部発動してポジションが反転する / キャンセル ack が
復路 latency を払わず queue position が楽観になる）。以下の数字はすべて**修正後**の再計測。

### コストの土俵（median spread, bps of mid, 2026-07-09）

maker-maker 往復の粗利上限は1スプレッド。手数料 1.5bps/側では **spread < 3bps の銘柄は
構造的に不成立**。taker フェーズの dev 3 銘柄がまさにそれ（9984 1.7bps / 5803 2.0 / 285A 2.6）。
50 銘柄中、6834 (21.9bps)・6368 (19.3)・3110 (14.2) など広スプレッド側にだけ余地がある。

### 検証結果: imbalance-gated passive entry は **全滅**（手数料 0 でも）

dev 6 銘柄（6834, 3110, 6269, 4062, 285A, 9984）で、enter×halflife×stop×hold×improve の
グリッド（join-touch 36 + spread 内改善 16 設定）を回した。**十分なサンプル（≥100 往復）を
持つ全設定が全 6 銘柄でマイナス**。最良でも join-touch -281 円/往復、improve=1 で -312 円/往復。

分解が示す死因は一貫している:

- **fill と予測失敗が同時に起こる**。imbalance が買い優勢のとき bid に並んでも、そこまで
  売りが到達する＝シグナルが外れた瞬間にだけ約定する。予測が当たると価格は約定なしで逃げる
  （missed fill）。9984 の v1 分解: spread capture +4.2k に対し taker_edge -21.3k、
  adverse -22.3k。しかも v1 の -113 円/往復は無条件ベンチマークの -114 円/往復と同じ——
  **シグナルゲートは per-fill 期待値を 1 円も改善しない**（取引回数を減らすだけ）。
- **taker 退出がスプレッドを焼く**。広スプレッド銘柄では stop/time-stop の成行が半スプレッド
  25 円/株を払い、獲得分 2〜3 往復を一撃で消す（6834: taker_edge -18k vs capture +7k）。
- **stop を外すと inventory drift に化けるだけ**（6834 no-stop: taker_edge -4k に減るが
  drift -29k に膨張）。損失は分解バケツ間を移動するだけで、符号は変わらない。
- **spread 内改善（improve）は fill を速くし adverse を減らすが、獲得も 1 tick 減り、
  ゼロには届かない**。

つまり retail latency (0.5s)・queue 最後尾・リベート無しという条件では、「板に置く」という
プリミティブ自体の per-fill 期待値がほぼ全域で負であり、シグナルゲート・価格改善・stop 設計は
その負をバケツ間で移すだけだった。

### 事前登録した判定

- 判定基準（実装前に凍結）: 手数料 0 で ≥100 往復/銘柄日の設定が dev 銘柄の 2/3 以上で
  net/trip > 0 → この family は不成立とする。
- 結果: **FAIL（family は死亡）**。最良設定（enter=0.3, halflife=20s, stop=5t, hold=300s,
  improve=1）を凍結して全 50 銘柄に展開した breadth check: **0/50 銘柄がプラス**、
  8,134 往復で -189 円/往復（手数料 0、fill rate 36.1%）。dev 銘柄の作為ではない。
- ただしこれは「imbalance-gated passive entry family」の死であり、maker 方向全体の死ではない。

### 何が変われば生き返るか

1. **latency**: 0.5s は市場の中で最後尾。コロケーション水準（ms 台）は前提から別物。
2. **歩み値**: volume 差分ではなく真の trade feed があれば queue 推定の悲観を緩められる
   （attribution 感度で確認済みだが、モデル悲観だけが敗因ではない — 手数料 0 でも gross 負）。
3. **戦略空間**: 寄り/引け auction 参加、昼休み再開の itayose、銘柄間 lead-lag、両面 MM の
   在庫スキューなど、本シミュレータで未検証の family は残っている。
   → **次期 3 family（auction / 両面 MM + inventory skew / lead-lag）の事前登録検証設計と
   kill criteria は [family-designs.md](family-designs.md) に凍結済み**（実装前に固定）。
4. **日数**: 確定データは 2026-07-09 の 1 日のみ。regime 依存の結論である可能性は排除できない。
   recorder が日数を積んだら、同じ凍結プロトコルで再評価する。
